Exploration Shell — UX Spec
System-level patterns common across all explorations. Spec + visual mockups.
Section 1
Opening Screen
  • Kid taps the exploration card → enters the exploration.
  • 3 components:
    • Heading — exploration title.
    • Context — image + text, child-friendly. Sets up the scenario and delivers the hook.
    • CTA — "Start" / "Dive in" / "Find out" — contextual per activity.
  • No estimated time shown.
  • Layout flow: Heading → Context → CTA. Concise, no clutter.

Section 2
Top Bar
  • Persistent throughout the exploration.
  • Layout: [←] ··· Title ··· [🔊] [3 of 8] [↻]
  • Left: ← Back button — goes to previous step. Hidden on step 1 / opening frame.
  • Center: Exploration title. Always visible.
  • Right:
    • 🔊 Mute/unmute toggle — persists across session.
    • Step progress — "3 of 8". Always show total.
    • ↻ Restart — restarts same activity. Triggers confirmation dialog.
  • Exit (X) is handled at system level by tech team — not part of the HTML.
  • All touch targets: min 2cm × 2cm.

Section 3
Step Structure
  • Linear only. No branching.
  • Progress bar shows major steps only. Sub-steps managed by HTML internally.
  • Step types (current set — expands over time):
Hook / Context
Sets up the scenario. No interaction.
Gut Check / Prediction
Kid commits an answer before seeing math.
Interactive Simulation
Kid manipulates something, sees the result.
Reveal / BIT
Shows the kid their gut was wrong.
Concept Unlocked
Names the math principle.
Tool Earned
Kid gets a reusable mental tool.
Summary / Reflection
Wraps up what they learned.
Section 4
Transitions
  • Animation: Fade-in + slight upward slide (0.45s ease). Same across all built explorations. Keep as-is.
  • "Next" button: Fixed at bottom. Label is contextual per step — "Let's find out", "Check your answer", "Next".
  • HTML → System data contract: WIP Define what HTML passes to app on each step transition.

Section 6
Answer Handling
  • Wrong: Subtle shake (6px, 0.4s) + soft pink highlight + encouraging text ("Not quite!" or "Almost!"). No red. No harsh sound.
  • After 2 wrong attempts → reveal the correct answer.
  • Right: Checkmark + brief positive text ("Nailed it!"). Not over-the-top.
  • Sound:
    • Wrong: gentle "oops" boop (400→350Hz sine, soft).
    • Right: rising two-tone (720→960Hz triangle).

Section 7
Back Navigation
  • ← button (top-left) — goes to previous step.
  • When reviewing past steps, → button appears — takes kid to the next step (forward through reviewed steps). Kid keeps pressing → to get back to their current step.
  • Past steps are read-only snapshots — inputs visible but locked. No animations replay.
  • Kid can go all the way back to step 1.
  • Bottom of past steps shows ← Prev and Next → navigation buttons for clarity.

Section 9
Exit Mid-Flow
  • Exit (X) handled at system level by tech team.
  • Saves at last completed step boundary.
  • If kid hasn't passed first interactive moment (not_started) → skip confirmation, just exit.
  • If in-progress → show confirmation dialog with step info.

Section 11
Ending Screen
  • Purpose: Closure. Show the kid what they learned and what they achieved.
  • Header: Activity-specific, written by HTML author per activity.
  • 1-line takeaway: Key learning from the activity (from HTML).
  • Performance:
    • XP — calculated by HTML, sent to system.
    • Accuracy — WIP Neha working on format.
    • Time spent — shown.
  • Share with Parents: Generates screenshot-style image (takeaway + XP/accuracy/time + Cuemath branding) → native share sheet. Lives here because this screen has the full picture of what the kid learned + achieved.
  • Celebration: Brief confetti (1.5s, fades) + celebrate sound.
  • Top bar: Back hidden, step pill shows "✓", restart hidden.
  • CTA: "Continue →" — goes to post-ending screen.

Section 12
Post-Ending Screen
  • Purpose: Rewards + Growth loop. Show what the kid earned and nudge them to bring friends.
  • Badge: TBD Visual badge/collectible earned for completing the activity. Badge system to be designed separately (one per activity? tiered? streaks?).
  • Invite a Friend: Growth loop CTA for PLD. Message: "I just explored [activity name] — try it!" + app/activity link → native share sheet.
  • Back to Home → Final exit. Only screen with this CTA.
  • No top bar — clean, full-screen feel. No back/restart.

  • Can tap ← to review previous steps, then → to move forward back to current step.
  • V2 1-line context reminder toast on resume (written per activity at build time).

Section 14
Error & Edge Cases
  • Offline: V1 requires internet. Show: "You need internet to start an exploration."
  • Slow load: Exploration thumbnail + title + "Getting your exploration ready..."
  • Crash recovery: Saves at step boundaries. Kid loses max ~2 min of in-step work. Resumes from start of the step they were on.
